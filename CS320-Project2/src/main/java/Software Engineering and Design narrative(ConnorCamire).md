This artifact is a Java project from my CS320 coursework. It's a contact management program built using Maven and JUnit. Uses a contact model that stores ID, phone number, and address, and a contact service that adds, retrieves, updates, deletes, and modifies using a hash map. The original version utilized unit tests, of course, but they were relatively basic. For this artifact milestone, I will be improving these unit tests under the software engineering and design category.

I chose this artifact because I wanted an artifact that was not utilizing Visual Studio and seemed valid for usage under all three categories, Given that I am using Linux on my main work machine, any assignments that utilize the single-file solution would be significantly more complicated and require unnecessary additional work in order to meet the requirements. And so in order to keep on the operating system I wish to use, I chose one of the Java projects that was made and identified that it could, for each category, this one in particular, be modified and improved, taking one simple category and adding much more to it. In this case, we'll be adding improved validation and error handling.

some specific examples of changes made in this artifact. Before the contact service method returned Boolean, true for success, false for failure. When called, it could not tell whether a failure was a duplicate ID, missing contact, or invalid field data. After the modification was made, We were given explicit exceptions, custom types for duplicate contact ID and contact not found were added. Separate files handle the logic for both of these, and they were implemented into tests so the tests can throw these new errors so that we know exactly what the issue was.

Originally, the validations focused mainly on null, checks, and string length. After the changes were made, we also rejected blank and white space values for ID, first name, last name, and address. Phone number must be exactly 10 digits.

Originally, service methods could be called regardless of whether they have a null or blank ID. Enhancements made it so that any operations that take a contact ID validate that the ID is not blank early during the process and will fail with a clear illegal argument exception separate from contact not found. This keeps invalid inputs and missing resources as two distinct failure methods, which would be useful for debugging since it grants more information to the tester than just error.

Originally, each update method—first name, last name, phone, address—repeated the same pattern: load contact, check existence, build new contact, store it. After, that pattern was centralized in a private helper so that four public update methods stay short and consistent. If validation or update rules change later, there is one place to adjust instead of four places that need adjustments.

Before, tests could confirm that an update returned success without providing the stored contact change. After the changes, tests use GetContactAfterUpdate to assert the actual field values in the hash map. New tests cover blank and null IDs, and existing tests cover duplicate, missing, and invalid update values.
